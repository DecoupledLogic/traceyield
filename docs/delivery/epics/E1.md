# Canonical usage store (MVP)

## Goal

Establish one **provider-blind canonical usage store** (`canonical.py` + a single
`usage.db`) that every coding-assistant provider ingests into as a neutral `Rec`
stream, and make it trustworthy enough to become the source of truth for the
report's metrics. The MVP is: two real providers ingesting, the report served
from the store, and storage that stays bounded over time.

## Definition of Done

- Both providers (Claude + Codex) ingest into one `usage.db`, distinguished by
  the `provider` column, with idempotent re-ingest.
- The report's daily/session aggregates are derived from the store (not a
  parallel dual-write path).
- Verbatim raw payloads age out on a retention window so the db stays bounded.

## Acceptance Criteria

- [ ] E1-F1 Multi-provider ingest complete (ClaudeProvider shipped; CodexProvider
      added and validated on the real corpus).
- [ ] E1-F2 Canonical-backed aggregation complete (metrics from `GROUP BY` over
      `usage.db`).
- [ ] E1-F3 Retention & storage hygiene complete (raw_event age-out).

## Test Plan

Rolls up the feature-level test plans; the epic is "done" when all three feature
stories are released and the full `python -m unittest` suite is green with both
providers exercised.

## Notes

First slice already shipped on `main` (commit `9a7fe2c`): schema, five neutral

Feature groups: **F1 ingest**, **F2 aggregation**, **F3 retention**.

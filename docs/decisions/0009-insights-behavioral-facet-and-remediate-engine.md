# Decision 0009: Adopt the Deterministic Best of Claude Code /insights (Behavioral Facet + Prescribe-to-Remediate Engine)

## Status

**Drafted - shaped from triage of request Q30. Not yet accepted.**

Generated by promoting intake request `Q30` (tempo-portfolio). Records the
governance statement "TraceYield should take the measured, honest parts of
`/insights` - a behavioral describe-facet and a prescribe-to-remediate engine -
and reject its inferred numbers and flattery." Not implemented until moved to
**Accepted** and planned into the `traceyield` project.

## Context

Claude Code's built-in `/insights` produces a working-style report over the same
transcripts TraceYield parses. A real run was captured and dissected in
[`../learning-from-claude-insights.md`](../learning-from-claude-insights.md). Two
findings frame this decision:

1. **`/insights` mixes measured counts with LLM judgments, and presents both as
   fact.** Its "Inferred Satisfaction" and "Outcomes" are model estimates rendered
   as counts; the same report's own "Where Things Go Wrong" documents that Claude
   fabricated time estimates the user had to correct. The counts are worth having;
   the inferred-numbers-dressed-as-counts and the sycophantic narrative are not.
2. **The substrate is already ours.** The behavioral signals `/insights` surfaces
   (response-time distribution, time-of-day activity, parallel sessions, tool and
   language mix, lines and files changed) are deterministic from transcripts
   TraceYield already reads. The error taxonomy already exists. What TraceYield
   lacks is the *behavioral facet* over that substrate and the *remediate rung* of
   its own describe -> diagnose -> predict -> prescribe -> remediate loop.

`/insights` also validates the loop's shape (it exercises all five rungs in one
pass) and shows exactly where TraceYield is weakest: it stops at "copy this into
Claude Code" and never applies anything.

## Decision

TraceYield will adopt two capabilities from `/insights`, both built only on
measured signals under the project's honesty doctrine.

**1. Behavioral describe-facet.** A second facet alongside the economic D-catalog,
computed deterministically from the transcript substrate and the existing error
taxonomy: response-time distribution (user think-time between turns), messages by
time-of-day, parallel-session overlap, tool/language/session-type mix, lines
added/removed and files touched, plus the B1-B5 behavioral ratios sketched in the
analysis doc.

**2. Prescribe-to-remediate engine.** Pattern-to-artifact rules gated on an
error/directive frequency threshold that emit ready-to-apply
CLAUDE.md/settings/hook/skill artifacts - each citing its count and an example
transcript - plus a `traceyield apply` subcommand that writes approved artifacts
into `.claude/` (idempotent, reversible, logged). This closes the loop `/insights`
leaves open.

**The honesty doctrine is a hard constraint.** No number is ever LLM-produced;
every claim is evidence-linked to a count and an example; the tool never
self-labels as an audit it did not perform; unknown means "unknown," never
invented.

**Explicitly out of scope** (recorded so planning does not reintroduce them):

- Any satisfaction or outcome number. Settled finding: the explicit user rating is
  not persisted in the transcripts TraceYield reads - `/insights` infers it from a
  model-estimated `user_satisfaction_counts` bucket, not the "How am I doing?"
  prompt. If Anthropic ever persists the real rating, revisit it as a legitimate
  facet.
- Semantic project clustering that requires an LLM. The deterministic `by_project`
  (by cwd) is the honest substitute; any named clustering must be labeled derived,
  never counted.
- Speculative "On the Horizon" narrative. Run-rate and runway (Decision 0003) are
  the honest predict rung.

## Consequences

- TraceYield gains a behavioral half of the loop to sit beside the economic half,
  and its first remediate-rung capability - a genuine differentiator over
  `/insights`, which only suggests.
- The `apply` engine writes real files into `.claude/`, so the honesty bar is
  higher than for a read-only report: artifacts must be idempotent, reversible, and
  logged, and no artifact may derive from an uncounted claim.
- The stylistic/workflow prescriptions `/insights` reaches by reading corrective
  user messages require a deterministic directive-detector (phrase-matching over
  user turns), not an LLM summarization pass - a bounded but real new parser.
- The runtime stays stdlib-only; `apply` fits the existing `cli.py` subcommand
  shape alongside `report`.

## Alternatives Considered

- **Mirror `/insights` wholesale, including the narrative.** Rejected: it imports
  the sycophancy and the inferred-numbers-as-fact failure mode the report itself
  demonstrates.
- **Behavioral facet only, no remediate.** Rejected as the stopping point: it
  leaves TraceYield where `/insights` already is - describing and prescribing but
  never closing the loop.
- **LLM narrative layer, clearly labeled.** Deferred: allowed only as a thin,
  evidence-required layer that never produces a number; not part of this decision's
  committed scope.

## Implementation Notes

Both capabilities are groundable in code that already exists
(`classification.py`, `cli.py`, the persisted metrics). Planning should decompose
this into (a) the behavioral facet as a reporting/rollup addition and (b) the
pattern-to-artifact rules plus the `traceyield apply` flow, under the guardrails in
the analysis doc.

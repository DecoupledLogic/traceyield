# T1: Implementation Report: CI/CD GitHub Actions

**Work item:** T1 (tech-debt): CI/CD GitHub Actions: run test suite on PRs and gate auto-merge on a required check
**Decision:** `inline:Q20` (SAS 3.00)

## What shipped

A single GitHub Actions workflow, `.github/workflows/ci.yml`:

- **Triggers** on `pull_request` (any branch) and `push` to `main`.
- **Job `ci`** on `ubuntu-latest`: `actions/checkout@v4` → `actions/setup-python@v5`
  (Python 3.12) → `python -m unittest test_report test_canonical`.
- **No dependency install, no build step**: the suite is stdlib-only and the
  tests parametrize their `root=`/`path=` seams to temp dirs, so the runner needs
  no `~/.claude`/`~/.codex` data, no secrets, and no network access.

Paired with **branch protection on `main`** requiring the `ci` status check, so
`gh pr merge --auto` waits on green CI instead of merging ungated, closing the
governance gap found in E1-F1-S1 (PR #2 auto-merged with no server-side checks).

## Acceptance criteria → evidence

| Scenario | Evidence |
|---|---|
| CI runs the suite on a PR | `on: pull_request` + `ci` job runs `python -m unittest test_report test_canonical`; check appears green on this PR. |
| CI runs on pushes to `main` | `on: push: branches: [main]`, same job/suite runs on main commits. |
| A failing test blocks auto-merge | `ci` registered as a **required status check** on `main` branch protection; `gh pr merge --auto` cannot complete until `ci` is green. This PR's own auto-merge is gated by it (positive proof). |
| No secrets or real user data | Verified locally on the worktree checkout: `Ran 102 tests … OK` with no pip install, no secrets, no `~/.claude`/`~/.codex`. |

## Decisions & scope

- **Single Python (3.12), `ubuntu-latest` only** per the Q20 scope hint (keep the
  matrix minimal). A `windows-latest` leg (mirroring the Windows-first
  `run.cmd`/Task Scheduler deploy target) is a noted possible follow-up, not in
  scope here.
- **Branch protection config:** requires the `ci` context; `enforce_admins=false`
  so Tempo's control-plane direct pushes to `main` (roadmap stamps) keep working;
  no required PR reviews; non-strict. The gate applies to PR merges, which is the
  governance surface the item targets.

## Follow-ups

- Optional `windows-latest` matrix leg.
- Optional: flip `enforce_admins=true` if the team later wants the gate to bind
  admins/direct-main-pushes too (would require routing roadmap stamps through PRs).

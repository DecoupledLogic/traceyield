# End-user guide set (docs/guides): install, use, update, troubleshoot, feature reference, cost-optimization playbook

## Goal

Give a **user** (not a contributor) everything they need to adopt traceyield and get lasting value from it: understand what it is, install and run it, read the report, keep it updated, fix problems, learn every feature in depth, and — the flagship — run the TraceYield loop themselves to cut spend without losing capability. Ships as a cross-linked `docs/guides/` set with a "start here" index, written for readers who have never seen the codebase.

## Definition of Done

- A `docs/guides/` folder exists with an index (`README.md`) and one focused guide per topic below, each self-contained and cross-linked.
- Every guide is grounded in the actual behavior of `report.py`/`canonical.py` and the emitted `report.html` (no invented features, no aspirational claims dressed as current behavior).
- The repo `README.md` links to the new guide set so users can find it.
- The existing test suite still passes (docs-only change must not break anything).
- Prose follows the repo conventions: **TraceYield** (brand) vs **traceyield** (tool/repo), no em dashes in shipped user-facing copy (mirrors E2-F5-S2), Windows-first but cross-platform commands.

## Acceptance Criteria

- [ ] **AC1 — Index.** `docs/guides/README.md` exists, states who the guides are for, and links every guide with a one-line "read this when…" hook, in a sensible reading order.
- [ ] **AC2 — Overview.** A guide explains what traceyield is, the describe -> diagnose -> predict -> prescribe -> remediate loop, what the report shows, and what it does/doesn't do (local-only, per-machine, no server), linking the framework doc for depth.
- [ ] **AC3 — Install.** A guide covers prerequisites (Python 3, stdlib-only), clone-and-run, where output lands (`machines/<id>/report.html`), per-machine data and `TRACEYIELD_MACHINE`, and platform notes (Windows / macOS / Linux).
- [ ] **AC4 — Using the report.** A guide walks a first run end-to-end: the summary line, opening `report.html`, and navigating every part of the dashboard (period/granularity switch, KPI cards + deltas, trend chart, breakdowns, tables), oriented around what a user *does* with each.
- [ ] **AC5 — Updating.** A guide covers keeping the tool current (`git pull`), what happens to durable stores on update, the merge/preserve semantics of `daily_metrics.json`/`session_metrics.json`, editing `PRICING` when rates change, and the "durable store has no git backup" sharp edge with a backup recommendation.
- [ ] **AC6 — Troubleshooting.** A guide gives symptom -> cause -> fix for the common failures: no/empty report, wrong machine folder, `python` vs `python3`, missing transcripts / rotated history, data-health warnings (schema drift, coverage holes, staleness), pricing-drift "skipped" notes, and scheduled-task issues.
- [ ] **AC7 — Feature reference.** A deep-dive reference documents each dashboard feature and its function: KPI cards, trend chart, cost-by-project/model/token-composition, top-sessions, routing-savings estimator, tokens-&-cost-per-tool, error taxonomy & fixes, pricing-over-time, and the data-health panel — including how each number is computed and its caveats (e.g. single-tool attribution exactness, routing estimate as upper bound).
- [ ] **AC8 — Cost-optimization playbook.** A guide teaches the diagnose -> prescribe -> remediate discipline: how to read cache-read-per-turn as context occupancy, how to spot runaway sessions, context bloat, model mis-routing, and tool-error waste, each paired with a concrete remedy (clear/compact context, `/model` routing, prompt/workflow fixes) and how to confirm the fix worked on the next report.
- [ ] **AC9 — Daily-run / automation.** A guide covers scheduling a daily run (Windows Task Scheduler via `run.cmd`, cron, launchd), the `PYTHON` override, multi-machine operation, and reading `run.log`.
- [ ] **AC10 — README link.** The repo `README.md` points readers to `docs/guides/` as the user-facing entry point.

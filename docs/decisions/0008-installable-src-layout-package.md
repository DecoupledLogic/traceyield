# Decision 0008: Repackage TraceYield as an Installable src-layout Package with a Stable CLI

## Status

**Drafted - shaped from triage of request Q29 (Codex Review). Not yet accepted.**

Generated by promoting intake request `Q29` (tempo-portfolio). Records the
governance statement "TraceYield should be an installable src-layout package with
a stable `traceyield` CLI, while keeping the runtime dependency-free" and the
phased direction. Not implemented until moved to **Accepted** and planned into the
`traceyield` project.

## Context

TraceYield is described as a "single-file tool", but that is no longer true in
practice: `report.py` is ~100 KB, `canonical.py` is ~35 KB, and the tests total
~120 KB. The growth has introduced coupling that a flat, script-oriented layout
no longer contains well:

- **A reverse dependency exists** - `canonical.py` imports helper functions from
  `report.py`, so the neutral canonical store depends on the reporting layer that
  should sit *above* it.
- **Distribution has no clean path.** Users clone the repo and run
  `python report.py`; there is no wheel, no console entry point, and the writable
  data (`machines/<id>/`, `usage.db`, reports, logs) lives *relative to the
  clone*. An installed package - especially under pipx - must never write into
  its own install directory, so packaging surfaces this assumption as a real
  constraint.
- **The HTML dashboard is a ~large Python string literal** (`HTML_TMPL`), which
  denies HTML/CSS/JS proper editor support even though it needs no templating
  engine.

The forces: we want cleaner internal boundaries and a supported cross-platform
install path (Decision 0005), **without** taking on a runtime dependency
(the stdlib-only promise is load-bearing for operational and security cost) and
**without** endangering the durable per-machine metrics store, which after the
history purge is the *only* copy of dates whose transcripts have since rotated
away (the "sharp edge" documented in CLAUDE.md).

Constraint that must be respected: this is a **structural migration**, not a
behavior change. It must not break `python report.py` for existing scheduled
tasks during the transition, and it must not move or recreate durable machine
data without an explicit migration + backup design.

## Decision

Migrate TraceYield to an **installable src-layout Python package** with a stable
console entry point (`traceyield = "traceyield.cli:main"`), keeping the **runtime
dependency-free** (build tooling may have dependencies; the installed runtime may
not). Deliver it as **five sequentially-gated phases**, each with an explicit exit
condition, never mixing reorganization with behavior redesign in one commit:

1. **Establish packaging without changing behavior.** Add `pyproject.toml` (PEP
   517 backend), create `src/traceyield/` with `__init__.py`, `__main__.py`, and
   `cli.py`; move implementations in with minimal internal change; leave **thin
   root compatibility wrappers** (`report.py`, `canonical.py`) so existing
   commands and scheduled tasks keep working; move tests under `tests/`; make CI
   install the package and run tests against the install; add smoke tests for
   `traceyield --help`, `traceyield --machine-dir`, and `python -m traceyield`.
   **Exit:** behavior + all tests pass from a checkout AND from an installed wheel.
2. **Fix dependency direction.** Extract path/config handling; extract shared
   pricing/classification; move canonical dataclasses into a neutral model
   module; stop canonical ingestion importing report; define a small provider
   interface/protocol; split Claude and Codex parsing into separate provider
   modules. **Exit:** providers depend only on neutral models/utilities; reporting
   depends on canonical data, not provider internals.
3. **Decompose reporting.** Separate aggregation, persistence, health, and HTML
   generation; move the HTML template into packaged `resources/report.html`
   loaded via `importlib.resources` (retaining the `__PAYLOAD__` / `__PRICEROWS__`
   markers); keep pure calculations isolated from file/network I/O; split the
   large test files along the same behavioral boundaries; centralize synthetic
   transcript fixtures in `tests/helpers.py`. **Exit:** changing dashboard
   presentation does not require editing ingestion/storage code, and vice versa.
4. **Make installation a supported product path.** Define the installed
   application's data-directory policy; design migration/compatibility for
   existing `machines/<id>` durable state **with a backup story**; add wheel +
   sdist builds to CI; test installation on Windows, macOS, and Linux;
   replace/supplement `run.cmd` with CLI-driven scheduling guidance; add release
   automation and artifact validation (including "the wheel contains the HTML
   resource" and "generated state is never written inside the installed
   package"). **Exit:** a clean machine can install, run, retain data across
   upgrades, and uninstall without risking user data.
5. **Remove transitional compatibility.** After at least one release: remove the
   root Python wrappers; update scheduled-task and documentation examples to
   `traceyield`; remove compatibility imports after confirming no consumers;
   tighten package-boundary and build-artifact tests.

The desired dependency direction is: `CLI -> ingestion -> canonical models/storage`
and `CLI -> reporting -> canonical models/storage`; provider adapters depend only
on shared models; pricing/paths/config are shared utilities. No provider or
canonical module imports the reporting layer.

## Consequences

### Positive

- The tool becomes installable and distributable (`traceyield`, `pipx install`),
  not just a git clone.
- The `canonical.py -> report.py` reverse dependency is eliminated; providers and
  reporting both sit above neutral models.
- The HTML/CSS/JS gets proper editor support as a packaged resource, with no
  templating engine and no build step.
- Writable-path handling becomes explicit and pipx-safe; the durable-data risk is
  handled deliberately (Phase 4) rather than implicitly through clone-relative
  paths.
- Test files split along behavioral boundaries; tests exercise the installed
  package, catching packaging errors early.

### Negative

- Multi-phase migration cost across the codebase and tests; sustained review
  discipline to keep reorganization and behavior change in separate commits.
- Transitional compatibility surface (root wrappers, compat imports) must be
  maintained for at least one release, then deliberately removed.
- Phase 4 carries real risk: the durable metrics store is the only copy of
  rotated-away history, so any data-directory relocation needs a migration +
  backup design before it ships.

### Neutral

- The stdlib-only runtime promise is unchanged; build/dev tooling (Hatchling or
  setuptools; optionally ruff/mypy/pytest) lives only in build or optional
  dev-dependency groups.
- `unittest` is retained as the test framework.

## Alternatives Considered

### Stay single-file (do nothing)

Keep `report.py` + `canonical.py` at the root and continue running via
`python report.py`.

**Why rejected:** the "single-file" framing is already fiction at ~100 KB +
~35 KB; the reverse dependency and clone-relative writable paths block clean
distribution and will only worsen. Doing nothing defers a growing cost.

### Big-bang reorganization + redesign in one change

Move everything into `src/`, fix the dependency direction, decompose reporting,
and change behavior together.

**Why rejected:** explicitly against the review guidance - mixing structural moves
with behavior redesign makes regressions unattributable and un-reviewable. The
phased plan with per-phase exit conditions is the safer path.

### Externalize the HTML with a template engine / web framework

Adopt Jinja or a framework to escape the giant string literal.

**Why rejected:** unnecessary - `importlib.resources` + the existing
`__PAYLOAD__` / `__PRICEROWS__` markers give editor support with zero runtime
dependency. A framework would violate the stdlib-only promise for no gain.

## Implementation Notes

- Do **not** move or recreate durable machine data automatically; Phase 4 owns the
  migration + backup design and is gated separately for that reason.
- Do **not** create a class per operation; most transforms stay plain functions.
  Do not split modules solely to make them short - split at responsibility
  boundaries.
- Do **not** make editable installs the only test path; test the built wheel.
- Preserve `report.py`'s `__PAYLOAD__` / `__PRICEROWS__` markers when the template
  moves to a packaged resource.

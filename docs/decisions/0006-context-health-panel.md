# Decision 0006: TokenLens Context-Health Panel

## Status

**Drafted — shaped from triage of request Q22. Not yet accepted.**

Generated by promoting intake request `Q22` (tempo-portfolio). Records the
governance statement "TokenLens should turn the token data it already collects
into diagnostic and prescriptive context-health signals, not just spend" and the
direction. Not implemented until moved to **Accepted** and planned in `traceyield`.

## Context

`report.html` today reports spend and routing, but not **context health**. The
token-mechanics doc (`docs/token-mechanics-and-insights.md`) defines a diagnostic
catalog **D1-D13** and a prescriptive playbook, and its §7 names a context-health
panel as the natural next build. The signals are ratios of fields **already
persisted** in `daily_metrics.json` / `session_metrics.json`
(`tok.{input,output,cache_read,cache_write_5m,cache_write_1h}`, `msgs`,
`tool_results`, `tool_errors`), so the MVP is a **client-side JS panel** mirroring
`aggregate()` / `renderRoute()` — **no parser change**.

The richer cuts (separating the static system+tools cache floor from conversation
growth; intra-session cost slope; retry-loop detection) want the turn-grained
canonical store (`usage.db`, epic **E1**). **E1 is now released**, so the phase-2
dependency is already satisfied — both phases are actionable.

## Decision

Add a client-side **"Context health" panel** to `report.html`, rendered per
selected period (day/week/month) from the existing payload, surfacing at least:

- **D1 context occupancy** (`cache_read / msgs`; avg tokens carried per turn) as a
  gauge framed against the model window.
- **D3 cache reuse** (`cache_read / (cache_write_5m + cache_write_1h)`) and/or
  **D4 write-premium burn** as a cache-efficiency readout.
- **D8 per-turn cost slope** and **D9 runaway-session** flags (high cost + high
  `cache_read/msgs`), tied into the existing Top-sessions table.

Each signal pairs with a one-line **prescriptive hint** (e.g. compact/clear, keep
the prompt prefix stable) consistent with the playbook. Deliver in two phases:
(1) the ratio panel now off aggregates; (2) floor-separation + intra-session slope
using the E1 turn-grained store.

## Consequences

- **Positive.** TokenLens becomes diagnostic and prescriptive, not just an
  accountant — higher product value from data already collected.
- **Cost.** Low — a JS addition to `HTML_TMPL` plus test-fixture math for the
  derived ratios (D1/D3/D8).
- **Risk.** Very low — no new capture, no parser change; the main care point is
  honest framing of gauges against the model window.

## Related

- Spec: `docs/token-mechanics-and-insights.md` (D1-D13, playbook)
- Epic **E1** (canonical store) — enables the phase-2 richer cuts (now released)

## Work Sizing (advisory, for planning)

Small feature, two phases. Phase 1: JS panel + `test_report.py` fixture math,
no Python capture change. Phase 2: floor-separation + intra-session slope off
`usage.db`. `/workflow-plan` off this decision in the `traceyield` project.

# Decision 0005: Package cc-usage-analytics as an Installable Cross-Platform Release

## Status

**Drafted — shaped from triage of request Q21. Not yet accepted.**

Generated by promoting intake request `Q21` (tempo-portfolio). Records the
governance statement "cc-usage-analytics should be an installable, distributable
release runnable on Windows and macOS" and the direction. Not implemented until
moved to **Accepted** and planned in the `traceyield` project.

## Context

Today `cc-usage-analytics` is a single-file stdlib Python tool (`report.py` +
`canonical.py`) with a **Windows-first deploy path**: a `run.cmd` wrapper driven
by Windows Task Scheduler, artifacts namespaced per machine under
`machines/<machine_id>/`. There is no packaged/installable release artifact —
users clone the repo and run `python report.py`.

The initial user base is a **product-development team split across Windows and
macOS**, so cross-platform parity is a first-class requirement, not a
nice-to-have.

**macOS unknowns to resolve at design.** The parser is stdlib Python and reads
`~/.claude/projects` / `~/.codex`, which exist on macOS, so the *core* likely
runs. The known gap is the **scheduled-run wrapper** (`run.cmd` + Task Scheduler
are Windows-only) — a macOS equivalent (launchd plist or cron) is the likely
work. Also confirm `machine_id()` (sanitized hostname, `TOKENLENS_MACHINE`
override) and path handling carry no Windows-only assumptions.

## Decision

Deliver a repeatable **release/packaging process** that produces an installable
artifact a teammate can install and run on **both Windows and macOS** — and, if
the tool does not already run on macOS, make it work there. The distribution
mechanism (pipx/pip package, standalone binary, Homebrew formula + winget/MSI,
or per-OS GitHub Releases artifacts) and whether the release ships the per-OS
recurring-run scheduler are **design decisions**, scoped here but not fixed.

Read the request's "a work flow that packages this" as a **deliverable release
capability**, not a Tempo `/workflow-*` skill.

## Consequences

- **Positive.** The tool becomes distributable beyond a git clone; the Win+Mac
  team can install and run it; macOS parity is established.
- **Cost.** New packaging/release process to build and maintain; macOS scheduler
  path to author and test; per-OS validation.
- **Risk.** Low-to-moderate — the core is stdlib and portable; the main unknowns
  are the scheduler equivalent and any latent Windows path assumptions.

## Related

- Epic **E1** (canonical store) and **T1** (CI/CD tech-debt) in this repo should
  be considered when planning — a packaging pipeline pairs naturally with CI/CD.

## Work Sizing (advisory, for planning)

Feature-scale. Likely: (a) macOS-parity audit + scheduler equivalent; (b) choose
+ implement the packaging mechanism; (c) per-OS release artifacts + install docs;
(d) CI wiring. `/workflow-plan` off this decision in the `traceyield` project.
